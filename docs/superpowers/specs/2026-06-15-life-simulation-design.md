# 南瓜生活模拟与关系分享设计

日期：2026-06-15

## 背景

当前项目已经有关系状态机、主动消息、社交模拟和 `life_events` 表。现有 `src/social/` 能生成角色相关的叙事弧事件，但产品体验上不应该让“虚拟社交”成为南瓜生活的全部。更合理的方向是：南瓜有自己的每日生活，社交只是生活的一部分。

本设计的体验目标是方案 C：关系越深，南瓜越自然地把自己的生活带进对话。实现切入口采用方案 B：先建立日常生活事件层，再让关系系统逐步接入。

方案 3（完整虚拟人生模拟器）很有吸引力，但不进入本期实现。它作为未来演进方向保留。

## 设计原则

1. 南瓜的生活不依附于用户每句话。用户消息只是检查机会，不是生活生成原因。
2. 生活事件进入聊天必须服务当前对话，不能为了分享而分享。
3. 大多数生活事件是低烈度日常。连续剧情和内心变化存在，但频率克制。
4. 当前轮回复最好只引用已经存在的生活痕迹，不临时编造“刚刚发生”的事件来贴合用户。
5. 一次聊天上下文最多注入 1 条生活事件，并允许 LLM 完全不提。
6. 提到生活事件后要回到用户，不把话题变成南瓜独白。
7. 用户对南瓜生活分享的回应会影响南瓜之后的分享频率、类型和关系倾向。

## 架构边界

新增 `src/simulation/` 作为生活层。它负责“南瓜今天发生了什么”，不负责完整重写关系状态机，也不替换现有社交模块。

建议模块：

- `life_scheduler.py`：统一入口，提供 `maybe_advance()`。判断是否该推进生活，大多数调用返回 0 条。
- `life_generator.py`：生成非社交生活事件，包括 `daily`、`creative`、`body_state`、`reflection`。
- `catchup_planner.py`：判断上次生活事件到现在是否有空白，必要时补 1-3 条低密度生活痕迹。
- `life_context_selector.py`：在 prompt 阶段从近期 `life_events` 选择 0-1 条可自然带出的事件。
- `life_sharing_policy.py`：判断哪些生活事件值得主动分享，且必须克制。
- `life_receptivity.py`：推断用户是否接得住南瓜的生活分享。
- `social_scheduler_adapter.py`：包装现有 `src/social.scheduler.Scheduler`，让社交事件成为生活层的一个事件来源。

现有 `src/social/` 本期不删除、不重写，只降级为生活事件来源之一。

## 生活事件类型

第一版使用现有 `life_events` 表，不新增数据库表。通过 `event_type` 和 `category` 约定语义。

事件倾向：

- `daily`：60%。普通日常，例如吃饭、上课、散步、拖延、刷手机、收拾东西、放空。
- `creative`：12%。创作/项目，例如写代码、想 prompt、做 bot、卡 bug、突然有灵感。
- `body_state`：15%。身体和能量，例如困、低电量、熬夜后迟钝、恢复了一点。
- `reflection`：8%。内心变化，例如深夜想法、自我怀疑、突然想通、情绪余波。
- `social`：5%。人物相关事件，继续复用现有 `src/social/` 的角色弧。

这些比例不是硬随机表，而是生成倾向。白天更容易 `daily/creative`，深夜更容易 `body_state/reflection`。`social` 保持低频，只有用户提到相关人物、已有活跃弧、冷却到期或时间触发时才推进。

连续性分三类：

- `standalone`：独立日常，没有链。
- `echo`：呼应近期事件，例如昨天卡住的 bug 今天想明白一点。
- `arc`：连续剧情节点，例如社交弧、长期项目或情绪问题推进。

第一版继续用 `life_events.causality_chain_id` 表达连续性。有 chain id 的事件视为连续事件，没有 chain id 的事件视为普通日常。

## 三条数据流

系统不采用“用户发消息 -> 必然生成生活事件”的线性数据流，而采用三条半独立的流。

### 聊天流

用户消息进入后，`ContextAssembler` 读取已有 `life_events`。`LifeContextSelector` 根据用户当前消息、关系、近期事件和用户接受度，选择 0-1 条生活事件注入 prompt。

注入不是命令，而是机会：

```text
南瓜最近有一件可联想到的生活小事：
- 下午有点低电量，出门买水时突然放空了十分钟。

如果自然相关，可以像朋友顺手一提；如果不贴合当前话题，就完全不要提。
不要解释这是记忆、事件或系统记录。
提到后要回到用户，不要展开成独白。
```

### 生活流

生活流由时间、空白、节律和上下文机会驱动。入口叫 `maybe_advance()`，而不是 `tick()`。

规则：

- 最近刚生成过生活事件时，返回 0 条。
- 到达普通推进间隔时，最多生成 1 条。
- 多天空白时，catchup 最多补 1-3 条。
- 用户提到某个角色或话题只提高相关事件概率，不保证生成。
- 本期集成方式是在每轮 postprocess 后调用 `maybe_advance()`；该方法必须先做冷却、概率和 due 判断，不能把“被调用”等同于“生成事件”。
- 后续可以增加真正的后台定时器。

### 分享流

已有生活事件先经过 `LifeSharingPolicy` 评分，再决定是否进入 `proactive_queue`。

主动分享条件：

- 事件值得分享，例如高情绪、有趣、连续节点、和用户长期关心的话题有关。
- 关系允许，通常 trusted 以上才考虑生活主动分享。
- 用户 `life_share_receptivity` 不低。
- 今天生活主动分享未超过 1 条。
- 刚聊完不久不主动发。
- 用户连续没回复时暂停。
- 发出后必须调用 `mark_event_shared()`，避免重复分享。

## 自然带入规则

硬规则不是“生活分享只能有三种用途”，而是：生活分享必须服务当前对话。

允许用途：

- 共情：用户说累、烦、开心，南瓜用自己的近况贴近对方。
- 类比：用户卡住、纠结、表达不清时，南瓜用自己的小事帮助表达。
- 轻转场：聊天变松、话题自然空出来时，顺手带一点近况。
- 被问起：用户问“你今天干嘛了”“你最近怎么样”时，可以直接讲。
- 连续性回应：用户之前知道或关心过某件事，南瓜可以补一句后续。
- 关系性自我暴露：关系够近时，南瓜偶尔说一点自己的状态，让对方感觉南瓜也在生活。
- 求意见/抛小问题：trusted 以上可以偶尔说“我今天有点卡在一个事上，你会怎么想？”

禁止：

- 用户正在倾诉时抢主场。
- 没有上下文硬塞“我今天发生了……”
- 把生活事件当日报。
- 每轮都提自己的事。
- 使用“系统记录”“life_event”“我被生成了一个事件”等元叙述。
- 提完后不回到用户。

## 关系与分享

关系状态不直接决定“能不能讲生活”，而是影响四件事：

- 分享频率：关系越近，越可能顺手提近况。
- 分享私密度：陌生人只适合轻日常，trusted 以后才适合低电量、反思、脆弱感。
- 表达方式：同一件事在 brother、respected、crush 下说法不同。
- 主动程度：关系越近越能主动，但主动永远比被动更克制。

可分享等级可以先运行时计算，不急着入库：

- `public`：轻日常，例如天气、散步、吃饭、出门。
- `casual`：有趣小事，例如犯蠢、卡住、突然想到什么。
- `personal`：状态变化，例如低电量、烦、拖延、自我怀疑。
- `vulnerable`：更脆弱的反思，例如关系焦虑、自我否定、深夜情绪。

关系门槛：

- `stranger`：只在被问起时分享 `public`。
- `acquaintance`：可自然带出 `public/casual`，不主动生活分享。
- `trusted`：可自然分享 `public/casual/personal`，主动分享需高分。
- `brother`：可以更随手和自嘲，但不能刷屏。
- `respected`：更偏反思、成长、求建议，表达克制。
- `crush`：可以更柔软和私密，但必须避免焦虑轰炸。

## Life Share Receptivity

新增概念 `life_share_receptivity`：用户对南瓜生活分享的接受度和参与度。

第一版先用轻量规则推断，不新增数据库字段。后续可持久化。

正向信号：

- 用户追问：“然后呢？”“后来怎么样？”
- 用户关心：“你还好吗？”“别太累。”
- 用户给建议：“你可以试试……”
- 用户接梗、互损、共鸣：“哈哈你也这样。”
- 用户主动问南瓜近况：“你今天干嘛了？”

负向信号：

- 用户无视南瓜分享，直接换话题。
- 用户明确说“不想听这个。”
- 用户对南瓜自我暴露反应冷淡或敷衍。
- 用户正在倾诉痛苦时，南瓜插入自己的生活导致话题偏移。

影响：

- `receptivity` 高：可更自然地偶尔提近况。
- `receptivity` 低：减少生活事件注入和主动分享，更多回到用户身上。
- 用户安慰或建议南瓜：增强 `trusted/respected` 倾向。
- 用户互损接梗：增强 `brother` 倾向。
- 用户暧昧式关心：增强 `crush` 倾向，但保持克制。

## MVP 范围

本期实现：

1. 新增 `LifeScheduler.maybe_advance()`，支持普通推进和 catchup 判断。
2. 新增 `LifeGenerator`，生成 `daily/creative/body_state/reflection`。
3. 新增 `LifeContextSelector`，在 prompt 中最多注入 1 条生活上下文。
4. 新增 `LifeSharingPolicy`，替代“有未分享事件就触发”的粗逻辑。
5. 新增 `LifeReceptivity` 规则推断，用于影响上下文注入和主动分享。
6. 新增 `SocialSchedulerAdapter`，复用现有 `src/social`。
7. 在 `ContextAssembler` 接入生活上下文选择。
8. 在 `PostProcessor` 中用生活层替代或包裹 `_social_tick()`。
9. 在主动消息路径中确保生活分享成功后调用 `mark_event_shared()`。

本期预留但不完整实现：

- `LifeState` 接口：为后续情绪、精力、压力、社交欲、创作欲预留。
- 后台真实定时器：第一版可先用 postprocess 后的 `maybe_advance()` 检查。
- 持久化 receptivity：第一版先规则推断，后续再入库。
- 更细的主动分享冷却：第一版只做每日 1 条和基础冷却。

本期不做：

- 不大改数据库 schema。
- 不删除 `src/social/`。
- 不重写 `branch_detector`。
- 不把 `relationship_type` 拆成复杂模型。
- 不做完整虚拟人生日程系统。
- 不做完整事件因果图。
- 不实现方案 3 的完整虚拟人生模拟器。

## 测试计划

`LifeScheduler`：

- 最近刚生成过 `life_event` 时，`maybe_advance()` 返回 0。
- 超过时间间隔时，最多生成 1 条普通事件。
- 多天空白时，catchup 最多补 1-3 条。
- 用户发消息不会必然生成生活事件。

`LifeGenerator`：

- 能生成 `daily/creative/body_state/reflection` 事件。
- 事件描述自然具体，非空。
- `daily` 是默认倾向。
- 深夜更容易生成 `body_state/reflection`。

`LifeContextSelector`：

- 不相关话题返回 0 条。
- 相关或情绪贴合时最多返回 1 条。
- `stranger` 不注入 `personal/vulnerable` 事件。
- 用户强倾诉时不抢话题。
- 已对该用户分享过的事件不重复注入。

`LifeSharingPolicy`：

- `stranger/acquaintance` 不主动分享生活。
- `trusted` 以上也必须过事件分数和冷却。
- 每天最多 1 条生活主动分享。
- 分享后调用 `mark_event_shared()`。
- `receptivity` 低时不主动分享。

`LifeReceptivity`：

- 用户追问、关心、接梗、问近况会提高接受度。
- 用户拒绝、无视、明显换话题会降低接受度。
- 接受度影响上下文注入和主动分享。

集成测试：

- `ContextAssembler` 能把生活上下文以“可选联想”的方式注入 prompt。
- `PostProcessor` 调用生活层不会让每轮消息都生成生活事件。
- 生活主动分享进入 `proactive_queue` 后成功发送，并标记已分享。
- 现有社交弧仍可生成 `social` 类型 life event。

## 未来迭代：方案三方向

未来可以把生活层扩展成完整虚拟人生系统，但需要分阶段推进：

- `LifeState`：持久化南瓜当前情绪、精力、压力、社交欲、创作欲。
- `Schedule`：上午、下午、晚上、深夜有不同生活节律。
- `LongTermGoals`：项目、学习、关系、自我成长的长期线。
- `CausalGraph`：生活事件之间互相影响，而不是只靠 `causality_chain_id`。
- `PersistentReceptivity`：长期记录每个用户是否愿意接住南瓜生活分享。
- `World Model`：角色网络、事件网络、生活空间、长期记忆共同组成虚拟人生层。
- `Adaptive Relationship`：关系不只由用户输入决定，也由南瓜自我暴露和用户回应共同塑造。

这些方向不进入 MVP，但当前模块边界应避免堵死它们。
